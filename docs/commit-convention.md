# 커밋 컨벤션 (Commit Convention)

이 문서는 본 프로젝트의 표준 커밋 메시지 작성 포맷과 모범 사례를 설명합니다.

## 1. 커밋 메시지 포맷

커밋 메시지는 반드시 **한글**로 작성해야 합니다.
기본적으로 첫 줄은 `type : title` 포맷을 따릅니다. 변경 사항이 너무 간단한 경우, 복잡한 양식 없이 "어떤 기능을 구현했다/수정했다" 정도로만 한글로 간단하게 설명해도 괜찮습니다.

단, 복잡하거나 공유할 맥락이 많은 중요한 변경 사항의 경우 아래와 같이 개요식(Problem, Changes, Implementation)으로 구체적인 내용을 작성하는 것을 권장합니다.

```text
type : title

(간단한 변경인 경우)
- 무엇을 구현/수정했는지 한글로 짧게 설명합니다.

(복잡한 변경인 경우 권장 포맷)
Problem
- 이 변경 사항을 왜(WHY) 만드는지 설명합니다.
- 해결하려는 문제나 요구사항이 무엇인지 적습니다.

Changes
- 무엇(WHAT)을 구현하거나 수정했는지 설명합니다.
- 추가된 기능이나 수정된 버그의 개요를 적습니다.

Implementation
- 어떻게(HOW) 변경 사항을 구현했는지 설명합니다.
- 기술적 접근 방식이나 주요 결정 사항을 상세히 적습니다.
```

### 허용되는 타입 (Allowed Types)
- `feat` : 새로운 기능 추가
- `fix` : 버그 수정
- `docs` : 문서 수정
- `style` : 코드 의미에 영향을 주지 않는 변경사항 (공백, 포맷팅, 세미콜론 누락 등)
- `refactor` : 버그 수정이나 기능 추가가 아닌 코드 구조 리팩토링
- `test` : 누락된 테스트 추가 또는 기존 테스트 수정
- `chore` : 빌드 프로세스나 문서 생성 같은 보조 도구 및 라이브러리 변경

### 예시 (Example)
```text
feat : 유저 권한 설정 슬라이더 추가

Problem
- 관리자 대시보드에서 권한 수준을 변경할 때 단순한 드롭다운만 있어서, 다시 0으로 초기화하기가 직관적이지 않았다.

Changes
- AccessLevel을 제어하던 Select 컴포넌트를 Slider 컴포넌트로 교체했다.

Implementation
- Ant Design의 Slider 컴포넌트를 사용하고 step 속성을 조정하여 정수 단위로 스냅되도록 구현했다.
- 프론트엔드의 `api/index.ts`에서 백엔드의 `access_level` 필드를 프론트엔드의 `accessLevel`에 매핑했다.
```

## 2. 커밋 안티패턴 및 모범 사례

**🚨 프로젝트 루트에서 절대 `git add .`를 무지성으로 사용하지 마세요! 🚨**

프로젝트 루트 디렉터리에서 `git add .`를 실행하여 모든 변경 사항을 한 번의 거대한 커밋으로 뭉뚱그려 올리는 것은 심각한 **안티패턴**입니다. 이는 코드 리뷰를 매우 어렵게 만들고, 커밋 히스토리를 오염시키며, 관련 없는 작업을 되돌리지 않고 특정 기능만 롤백하는 것을 불가능하게 만듭니다.

### 파일 스테이징 규칙
* **DO**: 변경 사항은 특정 파일이나 연관된 폴더 단위로 지정해서 커밋하세요. (예: `git add src/components/Slider.tsx src/api/index.ts`)
* **DO**: 커밋을 원자적(Atomic)으로 유지하세요. 하나의 커밋은 논리적으로 분리된 단 하나의 변경 사항만 담아야 합니다.
* **DON'T**: 프론트엔드 UI 변경 사항, 백엔드 API 추가, 문서 업데이트 등 서로 관련 없는 내용들을 묶어서 하나의 커밋으로 올리지 마세요.
